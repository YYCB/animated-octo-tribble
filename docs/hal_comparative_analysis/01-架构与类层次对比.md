# 01 — 架构与类层次对比

> **本章回答**:两家 Wrapper 从 `rclcpp::Node` 一路到 SDK,总共分了几层?每层的职责和接缝在哪里?HAL 应该从哪一层切入?

---

## 1. Orbbec 侧:三层 + 双业务节点

### 1.1 类层次

```
rclcpp::Node
     │
     ▼
OBCameraNodeDriver            ← 唯一 ROS 节点
 (设备管理器,Composable Node)
     │ owns
     ├──► OBCameraNode        ← 深度相机业务逻辑 (非 Node)
     │      │
     │      ▼
     │    ob::Pipeline ──► ob::Device
     │
     └──► OBLidarNode         ← LiDAR 业务逻辑 (非 Node)
            │
            ▼
          ob::Pipeline ──► ob::Device
```

### 1.2 三层职责划分

| 层 | 类 | 职责 | ROS 感知 |
|---|---|---|---|
| **Driver 层** | `OBCameraNodeDriver` (继承 `rclcpp::Node`) | 设备发现、热插拔、设备锁、固件更新、根据设备类型实例化 `OBCameraNode` 或 `OBLidarNode` | ✅ 是 Node |
| **Logic 层** | `OBCameraNode` / `OBLidarNode` | 传感器枚举、Profile 配置、帧回调、滤波、发布 Topic/Service/TF | ❌ 不是 Node,持有 `rclcpp::Node*` |
| **SDK 层** | `ob::Context / Device / Pipeline / Sensor / Frame / Filter` | OrbbecSDK v2 原生 API | — |

### 1.3 双业务节点设计 (关键特色)

根据 `device_info_->deviceType()` 动态分派:

```cpp
// OBCameraNodeDriver::initializeDevice()
auto device_type = device_info_->deviceType();
if (device_type == OB_STRUCTURED_LIGHT_MONOCULAR_CAMERA
    || device_type == OB_LIDAR) {
    ob_lidar_node_ = std::make_unique<OBLidarNode>(this, device, parameters_);
} else {
    ob_camera_node_ = std::make_unique<OBCameraNode>(this, device, parameters_);
}
```

**后果**:LiDAR 走 `publishScan`/`publishPointCloud` 路径,相机走 `publishFrame`/`publishPointCloud`/`publishRGBD` 路径,两者不共享业务类代码,但共享 Driver 层。

### 1.4 Composable Node 化

`OBCameraNodeDriver` 注册为 `rclcpp_components::register_node` 组件:
- 支持 `ros2 run` 独立运行
- 支持 `ComposableNodeContainer` 合并进程(多相机共享进程,减少 IPC)
- `use_intra_process_comms = true` 时实现零拷贝图像传输(`image_publisher` 封装层支持)

### 1.5 辅助类

| 类 | 作用 |
|---|---|
| `Parameters` | 动态参数管理 (薄封装,显式声明参数) |
| `image_publisher` | 封装 `image_transport` + intra-process |
| `D2CViewer` | 深度-彩色对齐可视化 |
| `FpsCounter` / `FpsDelayStatus` | 填充 `DeviceStatus` 消息 |
| `SyncedImuPublisher` | IMU 同步发布 |
| `JPEGDecoder` / `JetsonNVDecoder` / `RkMppDecoder` | 三平台解码器 |

## 2. RealSense 侧:工厂 + 单一业务节点 (可选 Lifecycle)

### 2.1 类层次

```
rclcpp::Node 或 rclcpp_lifecycle::LifecycleNode
     │  (由 USE_LIFECYCLE_NODE 宏决定)
     ▼
RosNodeBase  ← 类型别名
     │
     ▼
RealSenseNodeFactory          ← ROS 节点入口
 (设备发现/匹配/热插拔)
     │ owns
     ▼
BaseRealSenseNode             ← 核心业务逻辑 (非 Node)
     │ 持有 RosNodeBase& _node
     │
     ├──► RosSensor[]         ← 包装 rs2::sensor
     │      ├── SensorParams
     │      ├── ProfilesManager[] (Video + Motion)
     │      └── FrequencyDiagnostics
     │
     ├──► NamedFilter[]       ← 包装 rs2::filter
     │      ├── PointcloudFilter
     │      └── AlignDepthFilter
     │
     └──► rs2::device
```

### 2.2 RosNodeBase 的编译期条件切换

```cpp
#ifdef USE_LIFECYCLE_NODE
using RosNodeBase = rclcpp_lifecycle::LifecycleNode;
#else
using RosNodeBase = rclcpp::Node;
#endif
```

**意义**:RealSense 把 Lifecycle 作为构建选项内置。业务类 `BaseRealSenseNode` 始终用 `RosNodeBase&` 接口,对 Lifecycle 无感知。

### 2.3 "所有设备一个业务类"

与 Orbbec 的双业务节点相反,RealSense 不管 D435 / D455 / D457 / D585S 还是 D555,**都使用同一个 `BaseRealSenseNode`**。设备差异靠:
- **SDK 内部** (`librealsense2` 对每个 PID 处理)
- **设备能力探测** (运行时 `query_sensors()` + `is<safety_sensor>()` 判断功能)

例如安全相机只是通过如下探测加上额外的服务/话题:

```cpp
void BaseRealSenseNode::setSafetySensorIfAvailable() {
    auto iter = std::find_if(_dev_sensors.begin(), _dev_sensors.end(),
        [](rs2::sensor s) { return s.is<rs2::safety_sensor>(); });
    if (iter != _dev_sensors.end()) _safety_sensor = &(*iter);
}
```

### 2.4 `BaseRealSenseNode` 的多文件拆分

单个逻辑类,但实现分布在 ~6 个 cpp:

| 文件 | 大小 | 职责 |
|---|---|---|
| `base_realsense_node.cpp` | 55 KB | 帧回调、数据发布 |
| `rs_node_setup.cpp` | 30 KB | 初始化、话题创建 |
| `parameters.cpp` | 6 KB | 参数读取 |
| `tfs.cpp` | 13 KB | TF 坐标变换 |
| `safety.cpp` | 9 KB | D585S 安全相机 |
| `actions.cpp` | 6 KB | ROS2 Action |

**对比**:Orbbec 的 `ob_camera_node.cpp` 单文件达 ~220 KB。分文件有利维护,单文件聚合性更高。

### 2.5 传感器抽象 `RosSensor`

RealSense **让 `RosSensor` 直接继承 `rs2::sensor`**,从而沿用 SDK 语义:

```cpp
class RosSensor : public rs2::sensor {
    SensorParams _params;
    std::vector<std::shared_ptr<ProfilesManager>> _profile_managers;
    rs2::region_of_interest _auto_exposure_roi;
    // ...
};
```

Orbbec 侧没有等价的 `RosSensor` 类,`ob::Sensor` 的管理分散在 `OBCameraNode` 的成员 map 中:

```cpp
std::map<stream_index_pair, std::shared_ptr<ob::Sensor>> sensors_;
```

⚠️ **差异点**:RealSense 把传感器语义显式化为一个类(便于在 HAL 里抽成 `ISensor`);Orbbec 用按流索引的 map 管理。

## 3. 结构对比总览

```
                 Orbbec                                RealSense
┌─────────────────────────────────┐       ┌─────────────────────────────────┐
│  rclcpp::Node                   │       │  rclcpp::Node / LifecycleNode   │
│     │                           │       │     │                           │
│ ┌───▼────────────────────────┐  │       │ ┌───▼────────────────────────┐  │
│ │  OBCameraNodeDriver (Node) │  │       │ │  RealSenseNodeFactory      │  │
│ └───┬────────────────────────┘  │       │ └───┬────────────────────────┘  │
│     │                           │       │     │                           │
│ ┌───▼─────────┐ ┌─────────────┐ │       │ ┌───▼────────────────────────┐  │
│ │OBCameraNode │ │OBLidarNode  │ │       │ │  BaseRealSenseNode         │  │
│ │ (non-Node)  │ │ (non-Node)  │ │       │ │  (non-Node)                │  │
│ └──┬──────────┘ └──┬──────────┘ │       │ │   ├── RosSensor[]          │  │
│    │               │            │       │ │   └── NamedFilter[]        │  │
│ ┌──▼──────┐     ┌──▼──────┐     │       │ └───┬────────────────────────┘  │
│ │ob::     │     │ob::     │     │       │     │                           │
│ │Pipeline │     │Pipeline │     │       │     ▼                           │
│ └─────────┘     └─────────┘     │       │ librealsense2 (rs2::*)          │
└─────────────────────────────────┘       └─────────────────────────────────┘
```

| 维度 | Orbbec | RealSense |
|---|---|---|
| Node 层级 | 1 层 Node(Driver) + 2 种非-Node 业务类 | 1 层 Node(Factory) + 1 种非-Node 业务类 |
| 设备差异处理 | Wrapper 分化(相机 vs LiDAR) | SDK 内部屏蔽 |
| 传感器抽象 | map 散放 | `RosSensor : public rs2::sensor` |
| 滤波器抽象 | 散放在 `OBCameraNode` 成员中 | `NamedFilter` 基类 + 2 个子类 |
| 参数抽象 | `Parameters` | `Parameters` + `SensorParams` + `ParametersBackend` |
| Lifecycle | ❌ | ✅ (编译可选) |
| Composable Node | ✅ | ✅ (但主文档偏 standalone) |
| 单文件 vs 多文件 | 单文件 220KB | 6 文件 ~120KB |

## 4. 对 HAL 设计的可复用要点

### ✅ 共性 (可直接抽象)

1. **"Node 层 / Logic 层 / SDK 层" 三层划分**——两家业务类都不继承 Node,**HAL 的自然切入位置就是 Logic 层 ↔ SDK 层之间**。HAL 的客户是业务类(现在叫 `OBCameraNode` / `BaseRealSenseNode`,未来叫 `HalCameraNode`),而不是 ROS 节点本身。

2. **`stream_index_pair<type, index>` 作为统一流键**——Orbbec 的 `std::pair<ob_stream_type, int>` 与 RealSense 的 `std::pair<rs2_stream, int>` 语义完全一致。HAL 应提供:
   ```cpp
   struct StreamIndex { StreamType type; int index; };
   // 必须 operator==, operator<, hash, 以便做 map 的 key
   ```

3. **一个设备 → 多个传感器 → 每传感器多 Profile** 的三层资源模型。

### ⚠️ 差异点 (HAL 必须显式决策)

1. **业务类分化 vs 单一**:
   - Orbbec 靠 Wrapper 层的 `OBCameraNode` vs `OBLidarNode` 区分相机和 LiDAR。
   - RealSense 单一业务类,按 `Sensor` 能力探测来决定要发什么。
   - **HAL 建议采用 RealSense 模式**:`IDevice::getSupportedSensors()` 返回列表,业务层按需启停。LiDAR 作为一种 `SensorType::Lidar`,而非独立业务类。

2. **Sensor 抽象强弱**:
   - RealSense 有显式 `RosSensor` 类。
   - Orbbec 没有。
   - **HAL 必须定义 `ISensor` 接口**,对齐 RealSense 的粒度。

3. **Lifecycle 支持**:
   - RealSense 编译期可选。
   - Orbbec 无。
   - **HAL 本身不耦合 Lifecycle**,把 Lifecycle 作为 ROS Adapter 层的可选编译特性。

4. **业务类文件拆分策略**:
   - RealSense 按职责(setup / tfs / safety / actions / parameters) 分 6 文件是好实践。
   - **HAL 参考实现应采用多文件**,避免 Orbbec 单文件 220KB 的维护负担。

### 🎯 直接导出的接口 (见 [第 15 章](15-HAL核心接口详细设计.md))

```cpp
class IDeviceManager;          // 对应 OBCameraNodeDriver 设备管理部分 / RealSenseNodeFactory
class IDevice;                 // 对应 ob::Device / rs2::device
class ISensor;                 // 对应 RosSensor / OBCameraNode 中的 sensors_ map
class IFrame / IFrameSet;      // 对应 ob::Frame / rs2::frame
// HAL 使用者(业务节点)自己对应 OBCameraNode / BaseRealSenseNode 的角色。
```

### 架构反模式 (HAL 必须规避)

- ❌ 让 HAL 业务接口继承 `rclcpp::Node`——两家都没这么做,坚持非 Node 化。
- ❌ 让 HAL 暴露 `rs2::*` 或 `ob::*` 类型——这会让后端泄露。
- ❌ 让 Wrapper 层做设备类型分化(Orbbec 做法)——应统一由 `SensorType` 能力标记。
