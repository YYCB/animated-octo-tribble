# 04 — 数据流与帧回调机制对比

> **本章回答**:帧从 SDK 线程出来后怎么流转?什么时候在 SDK 线程直接处理?什么时候转到其他线程?帧同步(多流对齐)是由 SDK 做还是 Wrapper 做?HAL 的 `IFrame` / `IFrameSet` / `IFrameSyncer` 该怎么设计?

---

## 1. Orbbec 侧:主 Pipeline + 彩色队列线程

### 1.1 整体数据流

```
硬件
  │
  ▼
OrbbecSDK 内部 Pipeline
  │ (SDK 回调线程)
  ▼
onNewFrameSetCallback(FrameSet)
  ├── 提取 Color Frame → color_frame_queue_ (生产者)
  │                       │
  │                       └─ colorFrameThread_ (独立消费者线程)
  │                            └─ onNewColorFrameCallback()
  │                                 ├── softwareDecodeColorFrame()
  │                                 ├── processColorFrameFilter()
  │                                 └── onNewFrameCallback(COLOR) → publish
  │
  ├── 提取 Depth Frame
  │     └─ processDepthFrameFilter() → onNewFrameCallback(DEPTH) → publish
  │
  ├── 提取 IR Frame(s) → onNewFrameCallback(IR) → publish
  │
  └── publishPointCloud(frame_set)
        ├── publishDepthPointCloud()   # XYZ
        └── publishColoredPointCloud() # XYZRGB

独立的 IMU Pipeline:
  └── onNewIMUFrameCallback(frame) 或 onNewIMUFrameSyncOutputCallback(accel, gyro)
```

### 1.2 为什么彩色要独立线程?

MJPEG / H.264 解码是计算密集型,如果在 SDK 回调线程内同步解码,会导致:
- SDK 内部帧队列阻塞 → 掉帧
- 其他流(depth/IR)发布延迟

解决方案:**生产者-消费者队列**

```cpp
// 生产者: SDK 回调
std::unique_lock lock(color_frame_queue_lock_);
color_frame_queue_.push_back(color_frame);
color_frame_queue_cv_.notify_one();

// 消费者: colorFrameThread_
while (!exit_) {
    std::unique_lock lock(color_frame_queue_lock_);
    cv.wait(lock, [this](){ return !queue.empty() || exit_; });
    auto frame = queue.front(); queue.pop_front();
    lock.unlock();

    processAndPublishColorFrame(frame);
}
```

⚠️ 如果 queue 满了会怎样?Orbbec 限制队列深度(通常 3-5),满则丢老帧。

### 1.3 时间戳处理(3 种时间域)

```cpp
uint64_t getFrameTimestampUs(frame) {
    if (time_domain_ == "device")  return frame->timeStampUs();       // 硬件时钟
    if (time_domain_ == "system")  return frame->systemTimeStampUs(); // 主机时钟
    /* global */                   return frame->globalTimeStampUs(); // SDK 对齐后
}
```

### 1.4 帧同步:SDK 侧完成

`ob::Pipeline` 保证同一 FrameSet 中的 Color/Depth/IR 在硬件层面已时间对齐(同触发源或 SDK 插值)。Wrapper 不做帧同步。

### 1.5 IMU 独立 Pipeline

IMU 不进主 Pipeline,而是 `imu_pipeline_` 单独启动:
```cpp
imu_pipeline_ = std::make_shared<ob::Pipeline>(device);
imu_pipeline_->start(imu_config, imuFrameCallback);
```

详见第 8 章。

### 1.6 线程安全

| 锁 | 保护 |
|---|---|
| `device_lock_` (recursive_mutex) | 设备操作 |
| `point_cloud_mutex_` (recursive_mutex) | 点云生成(耗时长,避免并发) |
| `color_frame_queue_lock_` + cv | 彩色帧队列 |
| `cleaning_` (atomic_bool) | 防止热插拔时重复清理 |

Orbbec 暴露 `withDeviceLock()` 模板让外部 Service 回调安全访问设备:
```cpp
template<typename F>
auto withDeviceLock(F&& f) {
    std::lock_guard<std::recursive_mutex> lock(device_lock_);
    return f();
}
```

## 2. RealSense 侧:asyncer 条件异步 + 按需 frameset

### 2.1 整体数据流

```
硬件
  │
  ▼
librealsense2 Sensor (SDK 回调线程)
  │
  ├── 视频流 → frame_callback_function
  │    │
  │    ├── if (sync_frames || any_filter_enabled):
  │    │     _asyncer.invoke(frame)  [投递到同步器]
  │    │     └─ asyncer 内部线程输出 frameset
  │    │          └─ frame_callback(frameset)
  │    └── else:
  │          frame_callback(frame) [单帧直走]
  │
  └── 运动流 → imu_callback_function
       ├── imu_callback(frame) [独立发布]
       └── if (imu_sync_method != NONE):
             imu_callback_sync(frame) [软件融合为 united IMU]
```

### 2.2 `rs2::asynchronous_syncer`

librealsense2 提供的开箱即用帧同步器:
- 输入:各传感器独立帧
- 输出:时间对齐的 `frameset`(组合所有启用流)

**关键点**:只有在 `sync_frames` 打开或任何滤波器启用时才走 asyncer——因为滤波需要 frameset,且 sync_frames 用户显式想要对齐。否则直接发,性能更好。

### 2.3 核心 frame_callback 逻辑

```cpp
void BaseRealSenseNode::frame_callback(rs2::frame frame) {
    double frame_time = frame.get_timestamp();
    if (!_is_initialized_time_base)
        _is_initialized_time_base = setBaseTime(frame_time, time_domain);

    if (frame.is<rs2::frameset>()) {
        auto frameset = frame.as<rs2::frameset>();

        // 暂停 IMU 发布,避免与图像时间错乱
        if (_synced_imu_publisher) _synced_imu_publisher->Pause();

        // 应用滤波管线
        for (auto& filter : _filters)
            if (filter->is_enabled())
                frameset = filter->process(frameset);

        // 分发各帧
        for (auto it = frameset.begin(); it != frameset.end(); ++it) {
            auto f = *it;
            auto sip = make_pair(f.stream_type(), f.stream_index());
            auto t = frameSystemTimeSec(f);

            switch(f.stream_type()) {
                case RS2_STREAM_LABELED_POINT_CLOUD: publishLabeledPointCloud(f, t); break;
                case RS2_STREAM_OCCUPANCY:           publishOccupancyFrame(f, t); break;
                default:
                    publishFrame(f, t, sip, ...);
                    if (align_depth_enabled && sip == DEPTH)
                        publishFrame(f, t, ..., _depth_aligned_image_publishers);
                    if (rgbd_ready) publishRGBD(...);
            }
        }

        if (_synced_imu_publisher) _synced_imu_publisher->Resume();
    } else {
        // 单帧路径
        auto sip = make_pair(frame.stream_type(), frame.stream_index());
        publishFrame(frame, ..., sip, ...);
    }
}
```

### 2.4 时间基准

```cpp
bool setBaseTime(double frame_time, rs2_timestamp_domain time_domain) {
    if (time_domain == RS2_TIMESTAMP_DOMAIN_SYSTEM_TIME) {
        return true;  // 无需偏移
    }
    _camera_time_base = frame_time;
    _ros_time_base = _node.now();
    return true;
}

double frameSystemTimeSec(rs2::frame f) {
    double t = f.get_timestamp();
    if (time_domain == SYSTEM_TIME) return t / 1000.0;
    return (_ros_time_base - _camera_time_base + t) / 1000.0;  // 偏移到系统时间
}
```

### 2.5 RGBD 组合发布

特殊分支:当 `enable_rgbd=true` 且同时 `sync_frames=true` + `align_depth=true` + color + depth 启用:
```cpp
publishRGBD(color_frame, depth_frame, camera_info_color, camera_info_depth)
// 发布 realsense2_camera_msgs::msg::RGBD 消息
```

### 2.6 Intra-process 发布

RealSense 提供两种图像发布器:
- `image_rcl_publisher`:原生 `rclcpp::Publisher<sensor_msgs::msg::Image>`,支持 `rclcpp::IntraProcessMessageBridge` 零拷贝。
- `image_transport_publisher`:`image_transport::Publisher`,支持 `compressed`/`theora` 等插件。

编译宏切换或运行时参数选择。

## 3. 对比总览

| 维度 | Orbbec | RealSense |
|---|---|---|
| 主回调 | `onNewFrameSetCallback` (总是 frameset) | `frame_callback_function` (单帧或 frameset) |
| 帧同步来源 | SDK 内部 Pipeline | `rs2::asynchronous_syncer` (Wrapper 层使用) |
| 异步策略 | **彩色独立线程 + 队列**(针对解码耗时) | **按需 asyncer**(仅 sync_frames 或有滤波时) |
| 时间域 | 3 种: `device` / `system` / `global` | 2 种: `HARDWARE` / `SYSTEM_TIME` |
| 时间基准校正 | 直接用 timestamp | 计算 `_camera_time_base` 与 `_ros_time_base` 差 |
| RGBD 组合 | ❌(XYZRGB 点云替代) | **✅ 自定义 `RGBD.msg`** |
| Intra-process | `image_publisher` 封装 | **双发布器**可选 |
| 安全点云 | ❌ | **Labeled PointCloud / Occupancy** 特殊分支 |
| IMU 路径 | **独立 Pipeline** | 与视频 sensor 共用,独立回调函数 |

### ⚠️ 关键差异点

1. **谁做帧同步**:
   - Orbbec:SDK 内部,Wrapper 被动接收 frameset。
   - RealSense:SDK 提供 `asynchronous_syncer` 组件,Wrapper 主动选择是否启用。
   - **HAL 决策**:HAL `IFrameSyncer` 应该是**可插拔组件**。默认不启用(单帧直走),用户请求 synced 时才启用。

2. **异步粒度**:
   - Orbbec 专门为 color 建队列,因 MJPEG 解码慢。
   - RealSense 条件化全链路 asyncer,但 RealSense 不走 MJPEG。
   - **HAL 决策**:解码异步应为 `IDecoder` 的内部实现细节,不暴露到 `IFrameSyncer` 层。

3. **RGBD 组合**:
   - RealSense 有 `RGBD.msg` 一条消息包 color+depth+两套 info。
   - Orbbec 没有(需用 `message_filters::sync_policies` 在应用层自己做)。
   - **HAL 决策**:HAL 提供 `ISyncedFrameSet`,ROS Adapter 决定发不发 RGBD 组合消息。

4. **时间基准**:
   - RealSense 做了 "设备时钟 vs 系统时钟偏移" 校正,在 ROS 侧 `stamp` 是真实系统时间。
   - Orbbec 暴露 3 种模式让用户自选。
   - **HAL 决策**:`IFrame::getTimestamp()` 返回纳秒,`getTimestampDomain()` 标记 `HARDWARE` / `SYSTEM` / `GLOBAL`,校正逻辑在 ROS Adapter 层实现(不在 HAL 内)。

## 4. 对 HAL 的可复用要点

### 4.1 `IFrame` 基类接口

```cpp
enum class TimestampDomain {
    Hardware,   // SDK/设备原始时间戳
    System,     // 主机系统时钟
    Global,     // SDK 对齐后(OrbbecSDK globalTimeStampUs)
};

class IFrame {
public:
    virtual StreamProfile getProfile() const = 0;
    virtual uint64_t getTimestampNs() const = 0;
    virtual TimestampDomain getTimestampDomain() const = 0;
    virtual uint64_t getFrameNumber() const = 0;

    // 原始数据(只读视图,生命周期绑定 IFrame)
    virtual const void* getData() const = 0;
    virtual size_t getDataSize() const = 0;

    // 元数据
    virtual std::string getMetadataJson() const = 0;
    virtual bool hasMetadata(const std::string& key) const = 0;
    virtual int64_t getMetadataInt(const std::string& key) const = 0;
};

class IVideoFrame : public IFrame {
public:
    virtual int getWidth() const = 0;
    virtual int getHeight() const = 0;
    virtual int getBytesPerPixel() const = 0;
    virtual int getStride() const = 0;
};

class IMotionFrame : public IFrame {
public:
    struct Data { float x, y, z; };
    virtual Data getMotionData() const = 0;
};

class IFrameSet {
public:
    virtual size_t size() const = 0;
    virtual std::shared_ptr<IFrame> getFrame(StreamIndex) const = 0;
    virtual std::vector<std::shared_ptr<IFrame>> getAllFrames() const = 0;
};

using FrameCallback    = std::function<void(std::shared_ptr<IFrame>)>;
using FrameSetCallback = std::function<void(std::shared_ptr<IFrameSet>)>;
```

### 4.2 `IFrameSyncer` 可选组件

```cpp
class IFrameSyncer {
public:
    virtual void submitFrame(std::shared_ptr<IFrame>) = 0;
    virtual void setCallback(FrameSetCallback) = 0;
    virtual void start() = 0;
    virtual void stop() = 0;

    // 后端工厂
    static std::shared_ptr<IFrameSyncer> createDefault();         // Wrapper 实现(基于 queue+时间窗口)
    static std::shared_ptr<IFrameSyncer> createRealSense();       // 包装 rs2::asynchronous_syncer
    static std::shared_ptr<IFrameSyncer> createPassthrough();     // 什么都不做(Orbbec 已 SDK 内同步时用)
};
```

### 4.3 解码异步的封装(不暴露到外部)

```cpp
class IDecoderPipeline {
public:
    virtual void submit(std::shared_ptr<IFrame> raw_frame,
                        std::function<void(std::shared_ptr<IVideoFrame>)> done) = 0;
};

// 实现:
// - DefaultSyncDecoder: 同步解码,小图用
// - QueuedAsyncDecoder: Orbbec 风格,独立线程
```

HAL 使用者一般不直接碰,由 `ISensor::start` 内部配置。

### 4.4 线程安全的约束

HAL 必须在文档中明确:
- **`FrameCallback` / `FrameSetCallback` 可能在任意 SDK 线程调用**,用户自己负责同步。
- **`ISensor::start/stop` 不可在 callback 内调用**(可能死锁——与 Orbbec 的 `recursive_mutex` 实践一致,应采用非递归锁 + 文档约束,更不易写错)。
- 两次连续的 `start` 必须先 `stop`。

### 4.5 零拷贝考虑

HAL 尽量让 `IFrame::getData()` 返回 SDK 内部的指针,不做拷贝。SDK 帧对象的生命周期通过 `shared_ptr<IFrame>` 管理,引用一直到用户释放。

- RealSense:`rs2::frame` 内部是引用计数的,天然支持。
- Orbbec:`ob::Frame` 同样引用计数。
- HAL 实现应该在析构时释放 SDK 帧引用,不显式 memcpy。

⚠️ 陷阱:如果业务层要把 image 发到 ROS 而 ROS publisher 是异步的,要确保 `IFrame` 的生命周期足够长——Orbbec 和 RealSense 都处理了这一点,但 ROS Adapter 的设计不当会复发这个 bug。

### 4.6 时间基准校正放 ROS Adapter 层

```cpp
class RosTimestampConverter {
public:
    rclcpp::Time toRosTime(std::shared_ptr<IFrame> frame);
    // 内部:
    //   if domain == Hardware:
    //     若未初始化 base,设之;
    //     返回 ros_base + (frame_ts - camera_base)
    //   if domain == System:
    //     返回 frame_ts 直接转为 rclcpp::Time
};
```

HAL 不负责这个,保持纯粹。
