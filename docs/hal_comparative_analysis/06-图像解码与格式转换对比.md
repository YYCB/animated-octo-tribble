# 06 — 图像解码与格式转换对比

> **本章回答**:RGB 帧怎么从 SDK 传出来?何时需要解码?硬件平台(Jetson/RK)怎么用硬件解码器?格式如何映射到 ROS `sensor_msgs/Image::encoding`?HAL 的 `IDecoder` / `IDecoderFactory` 该怎么设计?

## 1. Orbbec 侧:多格式 + 多平台硬解

### 1.1 SDK 暴露的原始格式

| `OBFormat` | 说明 | 典型流 |
|---|---|---|
| `OB_FORMAT_MJPG` | Motion JPEG 压缩 | USB 2.0 彩色流 |
| `OB_FORMAT_YUYV` | YUV 4:2:2 | 彩色 |
| `OB_FORMAT_UYVY` | YUV 4:2:2(另一种排列) | 彩色 |
| `OB_FORMAT_NV21` | YUV 4:2:0 半平面 | 彩色 |
| `OB_FORMAT_NV12` | YUV 4:2:0 半平面 | 彩色 |
| `OB_FORMAT_RGB` | RGB888 | 彩色 |
| `OB_FORMAT_BGR` | BGR888 | 彩色 |
| `OB_FORMAT_BGRA` | BGRA8888 | 彩色 |
| `OB_FORMAT_Y16` | 16-bit 灰度 | 深度 / IR |
| `OB_FORMAT_Y8` | 8-bit 灰度 | IR |
| `OB_FORMAT_H264` | H.264 编码 | 网络彩色流 |
| `OB_FORMAT_H265` | H.265 编码 | 网络彩色流 |

### 1.2 格式 → ROS encoding 映射

```
OB_FORMAT_YUYV/UYVY/NV21/NV12 → FormatConvertFilter → "rgb8"
OB_FORMAT_MJPG                → JPEGDecoder          → "bgr8"
OB_FORMAT_H264/H265           → 硬件解码器           → "bgr8"
OB_FORMAT_RGB                 → 直接使用              → "rgb8"
OB_FORMAT_BGR                 → 直接使用              → "bgr8"
OB_FORMAT_Y16                 → 直接使用              → "mono16"
OB_FORMAT_Y8                  → 直接使用              → "mono8"
```

### 1.3 MJPEG 三平台解码策略

```
MJPEG Frame
    │
    ├── [Jetson] USE_NV_HW_DECODER
    │   └── JetsonNVDecoder
    │       ├── NvJPEGDecoder   (Jetson Multimedia API)
    │       ├── NvBufferTransform (颜色空间转换)
    │       └── 输出: BGR / RGB Mat
    │
    ├── [Rockchip] USE_RK_HW_DECODER
    │   └── RkMppDecoder
    │       ├── MPP Decoder (RK MPP API)
    │       ├── RGA 颜色转换 (librga) / libyuv
    │       └── 输出: BGR Mat
    │
    └── [通用平台] 软件解码
        └── JPEGDecoder (libjpeg-turbo)
            ├── tjDecompress2() → RGB / BGR
            └── 输出: BGR Mat
```

### 1.4 彩色帧的异步解码

因 MJPEG/H.264 解码耗时,Orbbec 把 color 独立到队列+线程(见 [第 4 章](04-数据流与帧回调机制对比.md) §1.2)。

### 1.5 `FormatConvertFilter`

对 YUYV/UYVY/NV21/NV12,不走压缩解码,而是走 SDK 内置的 `ob::FormatConvertFilter` 做色彩空间转换到 RGB。通常是 SIMD 优化。

### 1.6 IR MJPEG 解码

IR 也可能是 MJPEG 格式(有些设备):
```cpp
decodeIRMJPGFrame(frame)
  → JPEGDecoder → mono8 / mono16
```

### 1.7 畸变矫正(可选)

```cpp
if (enable_color_undistortion_) {
    cv::Mat K = ...;  // 内参
    cv::Mat D = ...;  // 畸变 k1,k2,p1,p2,k3
    cv::Mat new_K = cv::getOptimalNewCameraMatrix(K, D, size, 0);
    cv::undistort(image, result, K, D, new_K);
}
```

这是独立于格式解码的后处理步骤。

### 1.8 CMake 可选编译

```cmake
option(USE_RK_HW_DECODER "Enable Rockchip hardware decoder" OFF)
option(USE_NV_HW_DECODER "Enable NVIDIA hardware decoder" OFF)
```

编译期决定用哪个解码器,运行时无切换。

## 2. RealSense 侧:SDK 侧完成,Wrapper 无解码

### 2.1 SDK 暴露的原始格式(节选)

`librealsense2` 直接给出:
- `RS2_FORMAT_Z16` (深度)
- `RS2_FORMAT_Y8` / `Y16` (IR)
- `RS2_FORMAT_RGB8` / `BGR8` / `RGBA8` / `BGRA8` (彩色)
- `RS2_FORMAT_YUYV` / `UYVY`
- `RS2_FORMAT_RAW8` / `RAW10` / `RAW16` (原始 Bayer)
- `RS2_FORMAT_Y8I` / `Y12I` (交错双 IR,固件合并发送)

### 2.2 无 MJPEG / H.264

- RealSense 设备固件直出 YUYV/RGB/Z16 等原生格式,**不走 MJPEG 压缩**,也不走 H.264。
- 因此 Wrapper **完全不需要解码器**,只需做格式到 ROS encoding 的映射。

### 2.3 格式映射表

```cpp
// initializeFormatsMaps() 初始化
_rs2_to_ros_format[RS2_FORMAT_Z16]    = "16UC1";
_rs2_to_ros_format[RS2_FORMAT_Y8]     = "mono8";
_rs2_to_ros_format[RS2_FORMAT_Y16]    = "mono16";
_rs2_to_ros_format[RS2_FORMAT_RGB8]   = "rgb8";
_rs2_to_ros_format[RS2_FORMAT_BGR8]   = "bgr8";
_rs2_to_ros_format[RS2_FORMAT_RGBA8]  = "rgba8";
_rs2_to_ros_format[RS2_FORMAT_BGRA8]  = "bgra8";
_rs2_to_ros_format[RS2_FORMAT_YUYV]   = "yuv422_yuy2";
_rs2_to_ros_format[RS2_FORMAT_UYVY]   = "yuv422";
_rs2_to_ros_format[RS2_FORMAT_RAW8]   = "mono8";
_rs2_to_ros_format[RS2_FORMAT_RAW10]  = "mono16";
_rs2_to_ros_format[RS2_FORMAT_RAW16]  = "mono16";
// Y8I / Y12I 需要分离
```

### 2.4 `Y8I` / `Y12I` 交错双 IR 的拆分

特殊处理:RealSense D435 等设备可以把左右 IR 交错在同一帧里以节省带宽。Wrapper 需要拆分:
```cpp
// Y8I: 每像素 2 字节,一个字节 left,一个字节 right
for (int i = 0; i < w*h; i++) {
    left_image[i]  = data[2*i];
    right_image[i] = data[2*i + 1];
}
// Y12I: 12-bit × 2,需要 bit-level 拆分
```

### 2.5 GPU 加速(可选)

通过 `accelerate_gpu_with_glsl` 参数开启,SDK 内部会用 GLSL 着色器做:
- 颜色格式转换
- 点云生成
- 视差/深度转换

Wrapper 不直接接触 GPU,只是开关 SDK 的功能。

## 3. 对比总览

| 维度 | Orbbec | RealSense |
|---|---|---|
| 压缩格式 | **MJPEG / H.264 / H.265** | ❌ 无 |
| 解码器 | **3 种**:libjpeg-turbo / Jetson NV / RK MPP | ❌ 无需要 |
| 格式转换 | SDK `FormatConvertFilter` + Wrapper OpenCV 变换 | SDK 内部(含可选 GLSL) |
| 交错双 IR | ❌ | **`Y8I` / `Y12I`** 需拆分 |
| 畸变矫正 | Wrapper `cv::undistort` 可选 | 点云/对齐时 SDK 处理 |
| GPU 加速 | ❌ | `accelerate_gpu_with_glsl` |
| 嵌入式 SoC 支持 | **一等公民** | 依赖 libuvc / V4L2 |

### ⚠️ 关键差异

1. **解码器需求根源**:
   - Orbbec 设备(特别是 USB 2.0 带宽场景、网络设备)走压缩,**必须** 有解码器。
   - RealSense 只支持 USB 3.0+,直出原始格式。
   - **HAL 决策**:`IDecoder` 必须是 HAL 里的头等概念,不能省略。

2. **HW 解码器的可插拔**:
   - Orbbec 用编译期宏切换,决定静态链接哪一个。
   - **HAL 决策**:采用运行时策略——在 `IDecoderFactory` 里按平台自动选择,回退到软解。编译期选项作为加速,不作为唯一途径。

3. **格式枚举的爆炸**:
   - 合并两家的格式集合后,HAL 的 `PixelFormat` 枚举需要覆盖大约 20 种。
   - **HAL 决策**:枚举力求窄(只列实际被用到的),并给每个格式标注"是否需要解码"属性。

## 4. 对 HAL 的可复用要点

### 4.1 `IDecoder` 接口

```cpp
class IDecoder {
public:
    virtual std::string getName() const = 0;       // "libjpeg-turbo", "NV-Jetson", "RK-MPP"
    virtual bool isHardwareAccelerated() const = 0;
    virtual std::vector<PixelFormat> getSupportedInputFormats() const = 0;
    virtual PixelFormat getOutputFormat() const = 0;

    // 解码:同步 API;异步由调用方用线程池包装
    // 输出是外部分配的 buffer(避免拷贝)
    virtual bool decode(const void* input, size_t input_size,
                         int width, int height,
                         void* output, size_t output_capacity,
                         int& output_stride) = 0;
};
```

### 4.2 `IDecoderFactory` 接口

```cpp
class IDecoderFactory {
public:
    // 查询可用解码器(按平台能力)
    virtual std::vector<std::string> listAvailableDecoders(PixelFormat input_format) = 0;

    // 创建一个解码器实例
    virtual std::shared_ptr<IDecoder> createDecoder(
        PixelFormat input_format,
        const std::string& preferred_decoder = "") = 0;

    // 全局单例获取
    static IDecoderFactory& instance();
};
```

### 4.3 解码器注册(插件机制)

```cpp
// 每个解码器自注册
class JpegTurboDecoder : public IDecoder { /* ... */ };

static bool jpeg_turbo_registered = [](){
    IDecoderFactory::instance().registerDecoder("libjpeg-turbo",
        [](){ return std::make_shared<JpegTurboDecoder>(); },
        /* platforms */ { "linux", "windows", "mac" },
        /* formats */ { PixelFormat::MJPG });
    return true;
}();
```

编译时通过 CMake 开关控制 Jetson / RK 插件的编入:
```cmake
if(ENABLE_JETSON_DECODER)
    target_sources(camera_hal PRIVATE jetson_nv_decoder.cpp)
    target_link_libraries(camera_hal PRIVATE nvjpeg nvbufsurface)
endif()
```

### 4.4 解码器选择策略

```cpp
std::shared_ptr<IDecoder> pickDecoder(PixelFormat fmt) {
    auto& f = IDecoderFactory::instance();
    auto available = f.listAvailableDecoders(fmt);

    // 优先级: 平台特定 HW > 通用 HW > SW
    for (auto pref : {"NV-Jetson", "RK-MPP", "libjpeg-turbo"}) {
        if (std::find(available.begin(), available.end(), pref) != available.end())
            return f.createDecoder(fmt, pref);
    }
    return f.createDecoder(fmt);  // 任意
}
```

### 4.5 格式能力表

```cpp
struct FormatCapabilities {
    PixelFormat format;
    bool needs_decoder;
    bool is_compressed;
    int  bytes_per_pixel;        // 0 表示可变(压缩)
    std::string ros_encoding;    // "rgb8", "mono16", ...
};

static const std::map<PixelFormat, FormatCapabilities> kFormatTable = {
    { PixelFormat::Z16,   { PixelFormat::Z16,  false, false, 2, "16UC1" }},
    { PixelFormat::RGB8,  { PixelFormat::RGB8, false, false, 3, "rgb8"  }},
    { PixelFormat::MJPG,  { PixelFormat::MJPG, true,  true,  0, "bgr8"  }},
    { PixelFormat::H264,  { PixelFormat::H264, true,  true,  0, "bgr8"  }},
    { PixelFormat::Y8I,   { PixelFormat::Y8I,  true,  false, 2, "mono8" }},  // 拆分
    // ...
};
```

### 4.6 集成到 `ISensor::start` 的流程

```
ISensor::start(profiles, callback)
   │
   ├── 对每个 profile:
   │    └── 如果 profile.format 需要解码:
   │         decoder = IDecoderFactory::createDecoder(profile.format);
   │         wrap_callback = [decoder, cb](raw_frame){
   │             auto decoded = decoder->decode(raw_frame);
   │             cb(decoded);
   │         };
   │    └── 否则 wrap_callback = cb;
   │
   └── backend_sensor->start(profiles, wrap_callback);
```

### 4.7 注意点

- ⚠️ **IR 也可能 MJPEG**:Orbbec 某些设备 IR 也用 MJPEG,HAL 不能把解码限制在 color 流。
- ⚠️ **硬件解码器初始化开销大**:NvJPEGDecoder 初始化需要数百 ms,应在 `ISensor::start` 阶段就完成,避免首帧延迟。
- ⚠️ **GPU 显存碎片**:Jetson 上 NvBuffer 频繁分配会导致碎片,HAL 应使用 pool。
- ⚠️ **畸变矫正**不要放在解码器里——它是一个独立的 `IFilter`(见 [第 7 章](07-深度后处理滤波器对比.md)),避免职责混淆。
- ⚠️ **零拷贝 vs 解码**:解码必然拷贝,但解码后的 buffer 可以是 IDMA/预分配 pool,HAL 应支持用户提供 `output buffer`。
