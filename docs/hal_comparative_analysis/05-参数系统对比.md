# 05 — 参数系统对比

> **本章回答**:两家如何把 SDK 内的硬件选项(曝光/增益/白平衡/激光强度/...) 映射成 ROS 参数?参数变更怎么回调到 SDK?怎么做线程安全?**如何把 SDK 新增的选项自动暴露为 ROS 参数?** HAL 的 `IParameterProvider` 应该支持什么能力?

> **特别提示**:RealSense 的参数系统是整份对比里**最值得 HAL 直接借鉴**的子系统,后面会详细讲。

---

## 1. Orbbec 侧:显式声明 + 手工注册

### 1.1 参数层级

```
Launch 文件 (DeclareLaunchArgument) ← 优先级最高
    │
    ▼
YAML 配置 (config_file_path)
    │
    ▼
代码默认值 (constants.h / 硬编码)
    │
    ▼
最终参数
```

### 1.2 `Parameters` 类(薄封装)

```cpp
class Parameters {
public:
    // 声明参数并绑定回调
    template<class T>
    T setParam(std::string name, const T& initial,
               std::function<void(const rclcpp::Parameter&)> cb = nullptr);

    // 泛型版本,直接绑成员变量
    template<class T>
    void setParamT(std::string name, T& value, ...);

    // 运行时更新
    void setParamValue(std::string name, ...);

    void removeParam(std::string name);
};
```

### 1.3 典型使用模式

```cpp
// OBCameraNode::getParameters()
parameters_->setParam("color_width", color_width_, [this](const rclcpp::Parameter& p){
    color_width_ = p.as_int();
    // 通常需要重启 Pipeline
    restart_needed_ = true;
});

parameters_->setParam("enable_color", enable_color_, [this](auto& p){
    enable_color_ = p.as_bool();
    restart_needed_ = true;
});

parameters_->setParam("color_fps", color_fps_, ...);
// ... 对几十个参数逐一重复
```

### 1.4 参数规模

根据 Orbbec 文档统计:
- 设备参数(serial_number, usb_port, net_device_ip, uvc_backend, time_domain, device_preset, enable_ldp, ...): ~15
- 彩色流: ~10 (width/height/fps/format/auto_exp/exposure/gain/wb/...)
- 深度流: ~10
- IR 流: ~10 (x2 双目)
- IMU: ~6
- 点云: ~5
- 滤波器: **~30**(12 种滤波器 × enable + 若干参数)
- 同步: ~10
- **总计 100+**

### 1.5 大部分参数对应一次 Pipeline 重启

Orbbec 大多数参数变更通过"回调内设置 `restart_needed_` 标志 → 外部循环检测并 restart Pipeline" 的模式。

### 1.6 硬件选项直接走 Service

曝光/增益/白平衡/激光强度 等不走参数系统,而是走 Service(见 [第 11 章](11-ROS接口对比.md)):
- `/camera/set_color_exposure` (SetInt32)
- `/camera/get_color_exposure` (GetInt32)
- `/camera/set_color_auto_exposure` (SetBool)
- ...

**后果**:"调曝光" 要发 Service 调用,不能用 `ros2 param set`。

## 2. RealSense 侧:三层架构 + **SDK 选项自动映射**

### 2.1 三层架构

```
┌──────────────────────────────────────────┐
│ Parameters                                │
│  (顶层,统一管理器)                         │
│  - map<name, callback>                    │
│  - 更新线程 + deque<function>             │
└─────────────┬────────────────────────────┘
              │ owns
              ▼
┌──────────────────────────────────────────┐
│ ParametersBackend                         │
│  (底层,ROS 参数服务器封装)                 │
│  - rclcpp::Node 参数 API 封装             │
│  - on_set_parameters_callback             │
└──────────────────────────────────────────┘

┌──────────────────────────────────────────┐
│ SensorParams                              │
│  (传感器层,每 RosSensor 一个)             │
│  - 持有 shared_ptr<Parameters>           │
│  - registerDynamicOptions(rs2::options)  │   ← 核心亮点
└──────────────────────────────────────────┘
```

### 2.2 `Parameters` 类成员(更完整)

```cpp
class Parameters {
    RosNodeBase& _node;

    std::map<std::string, std::function<void(const rclcpp::Parameter&)>> _param_functions;
    std::map<void*, std::string> _param_names;           // 值指针 → 参数名 双向

    ParametersBackend _params_backend;

    std::condition_variable _update_functions_cv;
    bool _is_running;
    std::shared_ptr<std::thread> _update_functions_t;    // 更新线程
    std::deque<std::function<void()>> _update_functions_v;  // 更新队列
    std::list<std::string> self_set_parameters;          // 本节点自设参数
    std::mutex _mu;
};
```

**关键设计**:后台 **更新线程** + `deque<function>` 队列——当 ROS 参数服务器回调进来时,把 SDK 侧的 setter 函数 push 到队列,由线程异步执行,**避免在 ROS 回调线程中调用可能阻塞的 SDK 函数**(比如 `sensor.set_option()` 可能涉及 USB 控制传输,耗时数十 ms)。

### 2.3 `SensorParams::registerDynamicOptions` (⭐⭐⭐ 核心亮点)

```cpp
// 伪代码
void SensorParams::registerDynamicOptions(rs2::options& options,
                                          const std::string& module_name) {
    for (auto opt : options.get_supported_options()) {
        auto range = options.get_option_range(opt);
        auto option_name = rs2_option_to_string(opt);
        std::string param_name = module_name + "." + to_snake_case(option_name);

        // 根据 range.step 判断类型: int / float / bool / enum
        if (is_enum_option(opt)) {
            // 注册字符串参数,使用 option_value_description 枚举
            parameters->setParam<std::string>(param_name, default_value,
                [this, opt, &options](rclcpp::Parameter p) {
                    options.set_option(opt, parse_enum(opt, p.as_string()));
                });
        } else if (range.step == 1 && range.min == 0 && range.max == 1) {
            // bool
            parameters->setParam<bool>(param_name, default_value,
                [&](rclcpp::Parameter p){ options.set_option(opt, p.as_bool() ? 1 : 0); });
        } else if (is_integer_range(range)) {
            parameters->setParam<int>(param_name, default_value,
                [&](rclcpp::Parameter p){ options.set_option(opt, p.as_int()); });
        } else {
            parameters->setParam<double>(param_name, default_value,
                [&](rclcpp::Parameter p){ options.set_option(opt, p.as_double()); });
        }
    }
}
```

**意义**:
- SDK 新增的任意选项(例如新固件加了个 `emitter_frequency`),Wrapper **零代码改动** 即可自动暴露为 ROS 参数 `depth_module.emitter_frequency`。
- 用户可以 `ros2 param set /camera/camera/depth_module.emitter_frequency 2` 来改,不需要专门的 Service。
- 参数值与 SDK 选项的范围/步长/默认值由 SDK 报告,天然一致。

### 2.4 双向同步

```
方向 1: ROS → SDK  (用户 ros2 param set)
   ROS 参数服务器 → on_set_parameters_callback
      → Parameters::_update_functions_v 投递
         → 更新线程 → options.set_option(...)

方向 2: SDK → ROS  (硬件自动曝光导致值变化)
   定时轮询或 options.set_option_changed_callback (librealsense2 支持)
      → 比较当前 ROS 参数值与 SDK 当前值
         → 若不同,用 self_set_parameters 标记后 set_parameter
         → 不触发回调风暴(标记避免自反馈)
```

### 2.5 `self_set_parameters` 防循环技巧

```cpp
// 当 SDK 侧主动变化导致 Wrapper 要同步回 ROS
self_set_parameters.push_back(param_name);
node.set_parameters({rclcpp::Parameter(param_name, new_value)});

// on_set_parameters_callback:
if (std::find(self_set_parameters.begin(), end, name) != end) {
    self_set_parameters.remove(name);
    return success_without_invoking_sdk;   // 不再回调 SDK,避免循环
}
```

### 2.6 Profile 参数 vs Option 参数

RealSense 明确区分两类:
- **Profile 参数**(分辨率/fps/format):变更需要重启 sensor,由 `VideoProfilesManager` 手工注册。
- **Option 参数**(曝光/增益/...):变更不需重启,由 `registerDynamicOptions` 自动注册。

### 2.7 典型的 Launch 参数(节选)

```
rgb_camera.color_profile = "1280,720,30"
rgb_camera.color_format = "RGB8"
rgb_camera.enable_auto_exposure = true
rgb_camera.exposure = 100         # 自动映射自 RS2_OPTION_EXPOSURE
rgb_camera.gain = 64              # 自动映射自 RS2_OPTION_GAIN
rgb_camera.white_balance = 4600   # 自动映射自 RS2_OPTION_WHITE_BALANCE
rgb_camera.power_line_frequency = 1

depth_module.exposure = 8500
depth_module.gain = 16
depth_module.emitter_enabled = 1  # 自动映射自 RS2_OPTION_EMITTER_ENABLED

```

## 3. 对比总览

| 维度 | Orbbec | RealSense |
|---|---|---|
| 参数总数 | 100+(全手工) | 80+(Launch) + **SDK Option 自动映射** |
| 声明方式 | 手工逐个 `setParam` | Profile 手工,Option 自动 |
| SDK Option 暴露 | **走 Service**(`get_*` / `set_*`) | **走 Param**(`ros2 param set`) |
| 双向同步 | 有限(ROS → SDK) | **完整双向** |
| 参数变更线程 | 直接在 ROS 回调内执行 | **独立更新线程**(避免阻塞) |
| 防循环机制 | 无显式 | `self_set_parameters` 列表 |
| 运行时新选项 | 修改代码 + 重编 | **零代码**自动出现 |
| 粒度 | 粗(整机重启多) | 细(Option 级即时生效) |

### ⚠️ 本章最重要的差异

**RealSense 的 "SDK Option ↔ ROS Param 自动映射 + 异步更新 + 双向同步" 是行业标杆实践**,它在:
- 参数发现(用户 `ros2 param list` 就能看到所有硬件能力)
- 可维护性(SDK 升级无需改 Wrapper)
- 可脚本化(用 `ros2 param set` 做自动化测试)
- 性能(异步更新不阻塞 ROS callback 线程)

四方面都显著优于 Orbbec 的手工 Service 模式。

## 4. 对 HAL 的可复用要点

### 4.1 `OptionInfo` 结构

```cpp
enum class OptionType { Bool, Int, Float, Enum };

struct OptionInfo {
    std::string name;              // "exposure", "gain", "laser_power"
    std::string description;
    OptionType type;
    float min = 0, max = 0, step = 0, default_value = 0;
    bool is_readonly = false;
    std::map<std::string, int> enum_values;  // only for Enum
};
```

### 4.2 `ISensor` 的 Option API

```cpp
class ISensor {
public:
    // ...
    virtual std::vector<OptionInfo> getSupportedOptions() const = 0;
    virtual float getOption(const std::string& name) const = 0;
    virtual void setOption(const std::string& name, float value) = 0;

    // 变更事件(SDK → 上层)
    using OptionChangedCallback = std::function<void(const std::string& name, float value)>;
    virtual void setOptionChangedCallback(OptionChangedCallback) = 0;
};
```

### 4.3 `IParameterProvider` 抽象

HAL 本身不依赖 ROS,但 ROS Adapter 需要一个桥:

```cpp
// HAL 侧抽象(不依赖 ROS)
class IParameterProvider {
public:
    virtual void declareParameter(const std::string& name,
                                   const OptionInfo& info,
                                   float initial_value,
                                   std::function<void(float)> set_cb,
                                   std::function<float()> get_cb) = 0;

    virtual void declareParameter(const std::string& name,
                                   bool initial_value,
                                   std::function<void(bool)> set_cb) = 0;

    // ...重载 for int / string
};

// ROS 侧实现(在 ROS Adapter)
class Ros2ParameterProvider : public IParameterProvider {
    Parameters _params;  // 基于 RealSense 的 Parameters 类移植
    // ...实现异步更新线程、防循环、双向同步
};
```

### 4.4 自动映射的 Adapter 层逻辑

```cpp
// ROS Adapter
void registerSensorOptionsAsRosParams(
    std::shared_ptr<ISensor> sensor,
    const std::string& module_name,
    std::shared_ptr<IParameterProvider> provider)
{
    for (auto& opt : sensor->getSupportedOptions()) {
        std::string ros_name = module_name + "." + to_snake_case(opt.name);
        float current = sensor->getOption(opt.name);

        provider->declareParameter(ros_name, opt, current,
            [sensor, opt](float v){ sensor->setOption(opt.name, v); },
            [sensor, opt](){ return sensor->getOption(opt.name); });
    }

    sensor->setOptionChangedCallback([provider, module_name](
        const std::string& opt_name, float v) {
        std::string ros_name = module_name + "." + to_snake_case(opt_name);
        provider->updateParameterValue(ros_name, v);   // 触发 SDK→ROS 同步
    });
}
```

### 4.5 异步更新线程要点(移植自 RealSense)

```cpp
class ParametersAsyncBackend {
    std::thread _worker;
    std::deque<std::function<void()>> _queue;
    std::condition_variable _cv;
    std::mutex _mu;
    bool _running = false;

    void start() {
        _running = true;
        _worker = std::thread([this](){
            while (_running) {
                std::unique_lock lock(_mu);
                _cv.wait(lock, [this]{ return !_queue.empty() || !_running; });
                if (!_running) return;
                auto fn = std::move(_queue.front()); _queue.pop_front();
                lock.unlock();
                try { fn(); } catch (...) { /* log */ }
            }
        });
    }

    void submit(std::function<void()> fn) {
        std::lock_guard lock(_mu);
        _queue.push_back(std::move(fn));
        _cv.notify_one();
    }
};
```

### 4.6 参数 & Service 的职责切分

HAL 建议:
- **所有 SDK 硬件 Option → 统一走 ROS Param**(RealSense 模式)。
- **Service 仅保留**:
  - 跨流操作(`enable_rgbd`、`set_align_mode`)
  - 事件触发类(`trigger_capture`、`reboot_device`、`save_image`)
  - 查询类(`get_device_info`、`get_calibration_data`——因返回复杂结构)
- Orbbec 的 40+ 个 get/set Service 应该在 HAL 后端映射为参数,避免冗余。

### 4.7 参数命名规范

遵循 RealSense 模式:
```
{module_name}.{snake_case_option_name}
{module_name}.{stream}_profile     # Profile
enable_{stream}                    # 使能
```

### 4.8 特殊注意点

- ⚠️ **只读 Option**:两家 SDK 都有只读 option(如 asic_temperature),HAL 应标记 `is_readonly=true` 并拒绝 set。
- ⚠️ **参数持久化**:ROS2 Lifecycle 下 `configure → activate` 可能发生重声明,需要幂等。
- ⚠️ **并发更新**:用户连续 `ros2 param set` 多次,异步队列要能合并或顺序执行,避免 SDK 层竞争。
- ⚠️ **Enum Option 的字符串表示**:用 `"Preset.High"` 而不是 `2`,用户更友好。RealSense 做了,HAL 应该抄。
