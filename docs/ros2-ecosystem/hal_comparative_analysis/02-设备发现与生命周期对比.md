# 02 — 设备发现与生命周期对比

> **本章回答**:如何枚举设备?如何匹配目标设备?设备热插拔怎么响应?Lifecycle Node 的约束?HAL 的 `IDeviceManager` 接口应该支持哪些能力?

## 1. Orbbec 侧:Context + 定时器 + 回调 + 进程锁

### 1.1 设备发现入口

```cpp
// 1. 创建 SDK 上下文(单例概念,通常全局唯一)
ctx_ = std::make_unique<ob::Context>(config_path_);
ctx_->setLoggerSeverity(OB_LOG_SEVERITY_ERROR);

// 2. 注册设备变更回调
device_changed_callback_id_ = ctx_->registerDeviceChangedCallback(
    [this](std::shared_ptr<ob::DeviceList> removed,
           std::shared_ptr<ob::DeviceList> added) {
        onDeviceDisconnected(removed);
        onDeviceConnected(added);
    });

// 3. 查询设备
auto device_list = ctx_->queryDeviceList();
```

### 1.2 设备发现的 4 种渠道

```
ob::Context
  ├── USB 设备自动枚举 (默认,libuvc/v4l2 后端)
  ├── 网络设备枚举 (enumerate_net_device: true)
  │     └── ctx_->createNetDevice(ip, port)
  ├── GMSL 设备 (Jetson 平台)
  └── Force IP 配置 (通过属性控制)
```

### 1.3 设备选择策略(优先级自上而下)

```cpp
selectDevice(DeviceList list)
 ├─ if (!serial_number_.empty())     return selectDeviceBySerialNumber();
 ├─ if (!usb_port_.empty())          return selectDeviceByUSBPort();
 ├─ if (!net_device_ip_.empty())     return selectDeviceByNetIP();
 └─ return list[0];  // 自动选择第一个
```

### 1.4 初始化细节

```cpp
void initializeDevice(device) {
    device_info_ = device->getDeviceInfo();
    auto pid  = device_info_->pid();

    // 1. 时间同步
    if (enable_sync_host_time_) device->timestampReset();

    // 2. 固件更新
    if (!upgrade_firmware_.empty()) firmwareUpdate(upgrade_firmware_);

    // 3. 预设固件
    if (!device_preset_.empty())    device->loadPreset(device_preset_.c_str());

    // 4. 根据 device_type 实例化业务节点
    if (device_type == OB_LIDAR) {
        ob_lidar_node_ = std::make_unique<OBLidarNode>(...);
    } else {
        ob_camera_node_ = std::make_unique<OBCameraNode>(...);
    }
}
```

### 1.5 生命周期状态机

```
[未连接]
   │ 发现设备(onDeviceConnected)
   ▼
[连接中] device_connecting_ = true
   │ selectDevice() + initializeDevice()
   ▼
[已初始化] device_connected_ = true
   │ startStreams() / startIMU()
   ▼
[运行中] pipeline_started_ = true
   │ 发布图像/点云/IMU, 处理 Service, 设备状态监控
   │ 设备断开(onDeviceDisconnected)
   ▼
[断开中] clean() / 释放资源
   │ checkConnectTimer() 每秒检查
   ▼
[等待重连] ─────────────────────────────► [未连接]
```

### 1.6 进程锁(Orbbec 特色)

```cpp
pthread_mutex_t* orb_device_lock_;  // 共享内存 pthread_mutex
```

**用途**:当多个进程同时尝试打开同一台 Orbbec 设备时(例如一个系统有 3 个 camera 进程,一个错配 serial 就可能抢另一个设备),通过 **匿名共享内存上的命名 mutex** 互斥,防止 race condition 导致 SDK 初始化崩溃。这是生产环境长期运行中被证明必要的一个防护。

### 1.7 热插拔行为

```
USB 拔出 → SDK 内部检测 → onDeviceDisconnected
                             │
                             ├── cleaning_.store(true)
                             ├── 释放 ob_camera_node_ / ob_lidar_node_
                             ├── device_connected_ = false
                             └── 等待 checkConnectTimer 重新发现

USB 插入 → onDeviceConnected
            └── selectDevice + initializeDevice
```

### 1.8 关键参数

| 参数 | 默认值 | 说明 |
|---|---|---|
| `serial_number` | `""` | 目标序列号 |
| `usb_port` | `""` | 目标 USB 端口 |
| `net_device_ip` | `""` | 网络设备 IP |
| `net_device_port` | `0` | 网络设备端口 |
| `connection_delay` | `10` | 重连延迟(ms) |
| `enumerate_net_device` | `false` | 枚举网络设备 |
| `enable_heartbeat` | `false` | 心跳检测 |
| `retry_on_usb3_detection_failure` | `false` | USB3 检测重试 |
| `device_num` | `1` | 设备数量(用于多相机场景) |

## 2. RealSense 侧:Factory + Query 线程 + 可选 Lifecycle

### 2.1 设备发现入口

```cpp
class RealSenseNodeFactory : public RosNodeBase {
    rs2::device _device;
    std::thread _query_thread;
    // ...
};

void RealSenseNodeFactory::init() {
    _parameters = std::make_shared<Parameters>(*this);
    // 版本检查
    // Built with librealsense vX vs Running vX
    // 参数读取: _serial_no / _usb_port_id / _device_type / _wait_for_device_timeout / _reconnect_timeout / _initial_reset
    rs2::log_to_console(severity);
    _query_thread = std::thread([this](){ startDevice(); });
}
```

### 2.2 设备匹配维度

```cpp
struct DeviceMatchCriteria {
    std::string serial_no;        // 精确匹配
    std::string usb_port_id;      // 匹配 "4-2.4" 这样的物理端口路径
    std::string device_type;      // 正则匹配 (如 "d4.5[05]")
};
```

RealSense 支持 `device_type` 正则(例如 `d4.5[05]` 匹配 D435/D455),比 Orbbec 多这一维度。

### 2.3 设备事件 Callback

```cpp
_context.set_devices_changed_callback(
    [this](rs2::event_information& info) {
        // 1. 检查移除的设备
        for (auto& dev : info.get_removed_devices_list()) {
            if (_device && dev.get_info(RS2_CAMERA_INFO_SERIAL_NUMBER) == _serial) {
                onDeviceRemoved();
            }
        }
        // 2. 检查新增的设备
        for (auto& dev : info.get_new_devices_list()) {
            if (matchDevice(dev)) {
                onDeviceFound(dev);
            }
        }
    });
```

### 2.4 Lifecycle 节点状态机(可选)

当 `USE_LIFECYCLE_NODE` 定义时,`RealSenseNodeFactory` 继承自 `LifecycleNode`,状态机受 ROS2 Lifecycle 管理:

```
[unconfigured]
    │ on_configure() → 读参数、准备发布器
    ▼
[inactive]
    │ on_activate() → 开始设备查询、启动传感器
    ▼
[active]
    │ on_deactivate() → 停止传感器、保留发布器
    ▼
[inactive]
    │ on_cleanup() → 释放资源
    ▼
[unconfigured]
```

### 2.5 初始重置 `_initial_reset`

```cpp
if (_initial_reset) {
    _device.hardware_reset();
    // 等待设备重新枚举
}
```

用于工厂/开发机偶发固件抽风,强制让设备下电再上电。

### 2.6 `_wait_for_device_timeout` / `_reconnect_timeout`

两个超时用于控制阻塞式等待设备出现:
- **Wait timeout**:启动时,如果 -1 则无限等待
- **Reconnect timeout**:热插拔后的重连间隔

## 3. 对比总览

| 维度 | Orbbec | RealSense |
|---|---|---|
| SDK 上下文 | `ob::Context` (通常单例) | `rs2::context` + `RSContextSingletonWrapper` |
| 匹配维度 | `serial_number` / `usb_port` / `net_device_ip` | `serial_no` / `usb_port_id` / **`device_type`(正则)** |
| 网络设备 | **一等公民**(`createNetDevice(ip, port)`) | 不关注 |
| Force IP 配置 | ✅ | ❌ |
| 设备变更 | `registerDeviceChangedCallback` | `set_devices_changed_callback` |
| 热插拔回退 | 定时器 `checkConnectTimer`(每秒) | `_query_thread` 轮询 + `_reconnect_timeout` |
| 进程锁 | **共享内存 pthread_mutex** | ❌ |
| Lifecycle | ❌ | **编译期可选** |
| 初始重置 | `init_reset_` 参数控制 | `_initial_reset` 参数控制 |
| 固件更新 | **内置**(`firmwareUpdate(path)`) | ❌(需要 rs-fw-update 工具) |
| 设备预设 | `device->loadPreset(name)` | `json_file_path`(高级模式 JSON) |
| 超时配置 | `connection_delay` | `_wait_for_device_timeout` / `_reconnect_timeout` |

### ⚠️ 行为差异

1. **Context 单例**:
   - RealSense 使用 `RSContextSingletonWrapper`,多 Factory 共享一个 `rs2::context`,避免重复枚举。
   - Orbbec 每个 Driver 进程独立 `ob::Context`。
   - **HAL 决策**:应该像 RealSense 那样让 `IDeviceManager` 支持共享实例(由工厂方法控制),避免多实例开销。

2. **进程锁 vs 无锁**:
   - Orbbec 的共享内存进程锁在多进程多相机场景下是刚需。
   - RealSense 没有这一设计,依赖用户指定 serial 避免冲突。
   - **HAL 决策**:提供可选的 `IDeviceManager::enableProcessLock()`,后端决定是否支持。

3. **固件更新**:
   - Orbbec 内置到 Driver 中,参数 `upgrade_firmware: /path/to/fw.img` 即可。
   - RealSense 不做,外部工具处理。
   - **HAL 决策**:把固件更新作为可选能力 `IDevice::supportsFirmwareUpdate()`,接口保留,后端实现各异。

4. **设备能力匹配**:
   - RealSense 的 `device_type` 正则支持"我要任意一个 D4xx" 这种模糊匹配。
   - Orbbec 没有等价功能。
   - **HAL 决策**:`DeviceFilter` 应支持 `device_type_regex` 字段(后端按能力实现,不支持则返回 InvalidArgument)。

## 4. 对 HAL 的可复用要点

### 4.1 `IDeviceManager` 接口应覆盖的能力

```cpp
struct DeviceFilter {
    std::string serial_number;
    std::string usb_port;
    std::string ip_address;           // 网络设备
    int         net_port = 0;
    std::string device_type_regex;    // 模糊类型匹配
    int         device_index = -1;    // -1 = 自动
    std::chrono::milliseconds wait_timeout{-1};  // -1 = 无限
};

class IDeviceManager {
public:
    virtual std::vector<DeviceInfo> queryDevices() = 0;
    virtual std::shared_ptr<IDevice> openDevice(const DeviceFilter&) = 0;

    using DeviceChangedCallback = std::function<void(
        const std::vector<DeviceInfo>& added,
        const std::vector<DeviceInfo>& removed)>;
    virtual void setDeviceChangedCallback(DeviceChangedCallback) = 0;

    // 网络设备 (Orbbec 风格)
    virtual void enableNetworkDeviceEnumeration(bool) = 0;
    virtual std::shared_ptr<IDevice> connectNetworkDevice(
        const std::string& ip, int port) = 0;

    // 进程锁 (Orbbec 风格,后端可选)
    virtual void enableProcessLock(const std::string& lock_name) {}

    // 工厂方法
    static std::shared_ptr<IDeviceManager> create(const std::string& backend);
    // backend: "orbbec", "realsense", "auto"
};
```

### 4.2 `IDevice` 的生命周期 API

```cpp
class IDevice {
public:
    virtual DeviceInfo getInfo() = 0;

    // 硬件控制
    virtual void hardwareReset() = 0;     // 两家都有

    // 高级配置
    virtual bool supportsAdvancedMode() = 0;
    virtual void loadJsonConfig(const std::string& json) = 0;  // RealSense 风格
    virtual void loadPreset(const std::string& name) = 0;      // Orbbec 风格

    // 固件更新 (可选)
    virtual bool supportsFirmwareUpdate() = 0;
    virtual void updateFirmware(const std::string& path,
                                std::function<void(float)> progress_cb) = 0;

    // 时间同步
    virtual void syncHostTime() = 0;      // Orbbec 的 timestampReset()

    // 传感器
    virtual std::vector<std::shared_ptr<ISensor>> getSensors() = 0;
};
```

### 4.3 ROS Adapter 层的生命周期建议

- HAL 本身不耦合 Lifecycle。
- ROS Adapter 提供两个实现:`StandardRosAdapter`(`rclcpp::Node`) 和 `LifecycleRosAdapter`(`LifecycleNode`)。
- 编译开关同 RealSense:`USE_LIFECYCLE_NODE` 宏。

### 4.4 热插拔时业务类的销毁与重建

两家都采用 **"设备断开 → 销毁业务类 → 等重连 → 重建业务类"** 模式。HAL Adapter 层应遵循:

```cpp
void onDeviceAdded(DeviceInfo info) {
    _device = _device_manager->openDevice(_filter);
    _hal_camera_logic = std::make_unique<HalCameraLogic>(_node, _device);
    _hal_camera_logic->start();
}

void onDeviceRemoved() {
    _hal_camera_logic->stop();
    _hal_camera_logic.reset();
    _device.reset();
    // _device_manager 继续监听设备到来
}
```

### 4.5 陷阱提醒

- ⚠️ **回调线程安全**:两家的设备变更回调都来自 SDK 线程,业务类的销毁/重建必须加锁(Orbbec 的 `cleaning_` atomic 是典范)。
- ⚠️ **避免递归 Mutex 问题**:Orbbec 使用 `recursive_mutex` 管理设备锁,是因为 Service 回调可能在内部再调设备操作。HAL `ISensor` 实现需要考虑此约束。
- ⚠️ **硬件复位后**:设备 handle 失效,所有 Profile / Publisher 都必须重新获取;两家都有这个 bug-prone 的坑。
