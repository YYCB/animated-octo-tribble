# 09 — 多相机与硬件同步对比

> **本章回答**:多相机时怎么部署?设备之间如何避免冲突?如何实现"主-从硬件触发"或"软件触发"?HAL 的 `ISyncManager` 应该提供哪些能力?

## 1. Orbbec 侧:7 种同步模式 + 进程锁

### 1.1 多相机部署

**方式 1:多进程(推荐)**
```
┌─────────────┐  ┌─────────────┐  ┌─────────────┐
│ Process 1   │  │ Process 2   │  │ Process 3   │
│ camera_1    │  │ camera_2    │  │ camera_3    │
│ SN: AAA     │  │ SN: BBB     │  │ SN: CCC     │
└─────────────┘  └─────────────┘  └─────────────┘
       │              │                │
       └──────────────┼────────────────┘
                     USB
```

每个进程只通过 `serial_number` 锁定一台设备,通过 `orb_device_lock`(共享内存 pthread_mutex)防止多个进程同时打开同一设备。

**方式 2:组合容器(低延迟,intra-process)**
```
ComposableNodeContainer
  ├── OBCameraNodeDriver(camera_1, SN=AAA)
  ├── OBCameraNodeDriver(camera_2, SN=BBB)
  └── OBCameraNodeDriver(camera_3, SN=CCC)
```

进程内通过 ROS2 intra-process 直接传递 `sensor_msgs::msg::Image`,延迟更低。`use_intra_process_comms=true`。

### 1.2 Launch 配置

```python
for cam in [{'name':'camera_1','serial':'AAA'},
            {'name':'camera_2','serial':'BBB'}]:
    nodes.append(IncludeLaunchDescription(
        PythonLaunchDescriptionSource('gemini_330_series.launch.py'),
        launch_arguments=cam.items()))
```

### 1.3 7 种 `OBMultiDeviceSyncMode`

| 模式 | 说明 |
|---|---|
| `FREE_RUN` | 自由运行(默认,无同步) |
| `STANDALONE` | 独立模式(忽略外部触发) |
| `PRIMARY` | 主设备,提供触发信号 |
| `SECONDARY` | 从设备,被动接收触发,可能掉帧 |
| `SECONDARY_SYNCED` | 从设备,接收触发后保证同步输出 |
| `SOFTWARE_TRIGGERING` | 软件触发模式(由 ROS Service 触发) |
| `HARDWARE_TRIGGERING` | 硬件触发模式(GPIO/GMSL 等外部源) |

### 1.4 时序参数(细粒度)

```yaml
sync_mode: "primary"                    # 上述 7 种之一
depth_delay_us: 0                       # 深度帧触发延迟(μs)
color_delay_us: 0                       # 彩色帧触发延迟(μs)
trigger2image_delay_us: 0               # 触发到出图延迟
trigger_out_delay_us: 0                 # 触发输出延迟
trigger_out_enabled: true               # 触发信号输出使能
frames_per_trigger: 1                   # 每次触发产生几帧
```

### 1.5 主从配置示例

```yaml
# 主设备(Primary)
sync_mode: "primary"
trigger_out_enabled: true
trigger_out_delay_us: 0

# 从设备(Secondary)
sync_mode: "secondary"
depth_delay_us: 0
color_delay_us: 0
trigger2image_delay_us: 0
```

### 1.6 软件触发 Service

```
/camera/trigger_capture (std_srvs::srv::Trigger)
  └── 当 sync_mode=SOFTWARE_TRIGGERING 时,调用此 service 触发一次拍照
```

### 1.7 GMSL 触发(Jetson 平台)

```yaml
# Femto Mega 等 GMSL 设备
gmsl_trigger_fps: 30
trigger_pulse_width_us: 100
```

### 1.8 Interleave 模式

`enable_interleave_frame: true` 让设备在主曝光与副曝光之间交错(用于 HDR 或激光开/关交替)。

### 1.9 进程锁实现

```cpp
pthread_mutex_t* orb_device_lock_;

// 初始化:打开/创建共享内存
int fd = shm_open("orb_device_lock", O_CREAT | O_RDWR, 0666);
ftruncate(fd, sizeof(pthread_mutex_t));
orb_device_lock_ = (pthread_mutex_t*)mmap(...);

// 配置为进程间共享
pthread_mutexattr_t attr;
pthread_mutexattr_init(&attr);
pthread_mutexattr_setpshared(&attr, PTHREAD_PROCESS_SHARED);
pthread_mutex_init(orb_device_lock_, &attr);

// 使用
pthread_mutex_lock(orb_device_lock_);
selectAndOpenDevice();
pthread_mutex_unlock(orb_device_lock_);
```

## 2. RealSense 侧:简单同步 + 多 Launch

### 2.1 多相机部署

通过 `rs_multi_camera_launch.py`:
```python
for serial in ['SN1', 'SN2', 'SN3']:
    nodes.append(Node(package='realsense2_camera',
                      executable='realsense2_camera_node',
                      parameters=[{'serial_no': serial}, ...]))
```

或用 `rs_multi_camera_launch_sync.py` 加入硬件同步配置。

### 2.2 同步模式 (`inter_cam_sync_mode`)

```
0 = Default      # 不参与同步
1 = Master       # 主设备,提供 1Hz 同步信号(取决于 fps)
2 = Slave        # 从设备
3 = Full Slave   # 完全从设备(D455 等支持)
```

显著少于 Orbbec 的 7 种,缺少软件触发、GMSL 触发等模式。

### 2.3 帧同步(单设备内部多流)

参数 `enable_sync = true` 启用 `rs2::asynchronous_syncer`,把多流帧组合为 frameset。这是**单设备内**的同步,不是设备间。

### 2.4 时序参数

无类似 Orbbec 的细粒度延迟参数,RealSense 主要靠固件默认行为。

### 2.5 进程锁

❌ 无。多进程访问同一设备会产生 SDK 层错误,没有自动互斥。

### 2.6 D555 / DDS 网络相机

D555 通过 DDS 协议直接发布 ROS topics,不通过 USB。多相机可通过网络分布,使用 ROS2 DDS 自带的发现机制。

## 3. 对比总览

| 维度 | Orbbec | RealSense |
|---|---|---|
| 多进程支持 | ✅ + 进程锁 | ✅ 但无进程锁 |
| Composable 容器 | ✅ | ✅(但文档少) |
| 同步模式数 | **7** | 4 |
| 软件触发 | **✅** Service | ❌ |
| 硬件触发参数 | **细粒度**(6+ 延迟参数) | 仅 master/slave 切换 |
| GMSL 平台 | ✅ | ❌ |
| Interleave | ✅ | ❌ |
| 网络多相机 | ✅(自定义网络设备) | ✅(D555 DDS) |
| 单设备帧同步 | SDK 内部完成 | `enable_sync` + asynchronous_syncer |
| 进程锁 | **✅** 共享内存 mutex | ❌ |

### ⚠️ 关键差异

1. **同步能力**:Orbbec 远超 RealSense。RealSense 只能做"开/关同步",Orbbec 可以做完整的主从触发链路、软件触发、GMSL 触发。
2. **进程锁**:Orbbec 在多进程多相机场景下的可靠性更高。
3. **HAL 决策**:`ISyncManager` 应以 Orbbec 的能力为上界设计,RealSense 后端只实现子集(返回 NotSupported)。

## 4. 对 HAL 的可复用要点

### 4.1 `SyncMode` 枚举

```cpp
enum class SyncMode {
    FreeRun,            // 自由运行
    Standalone,         // 独立(忽略外部触发)
    Primary,            // 主设备
    Secondary,          // 从设备(可能掉帧)
    SecondarySynced,    // 从设备(保证同步)
    SoftwareTrigger,    // 软件触发
    HardwareTrigger,    // 硬件触发
};

struct SyncConfig {
    SyncMode mode = SyncMode::FreeRun;
    int depth_delay_us = 0;
    int color_delay_us = 0;
    int trigger2image_delay_us = 0;
    int trigger_out_delay_us = 0;
    bool trigger_out_enabled = false;
    int frames_per_trigger = 1;
};
```

### 4.2 `ISyncManager` 接口

```cpp
class ISyncManager {
public:
    virtual ~ISyncManager() = default;

    // 能力查询
    virtual std::vector<SyncMode> getSupportedSyncModes() const = 0;

    // 配置
    virtual void setSyncConfig(const SyncConfig&) = 0;
    virtual SyncConfig getSyncConfig() const = 0;

    // 软件触发
    virtual bool supportsSoftwareTrigger() const = 0;
    virtual void triggerOnce() = 0;

    // 状态
    virtual bool isSynced() const = 0;     // 同步链路工作正常?
};

class IDevice {
    virtual std::shared_ptr<ISyncManager> getSyncManager() = 0;
};
```

### 4.3 Backend 实现矩阵

```
                       Orbbec  RealSense
FreeRun              ✓        ✓
Standalone           ✓        ✗ (映射为 Default)
Primary              ✓        ✓ (Master)
Secondary            ✓        ✓ (Slave)
SecondarySynced      ✓        ✓ (Full Slave)
SoftwareTrigger      ✓        ✗ (NotSupportedException)
HardwareTrigger      ✓        部分(Master/Slave 实质是硬件同步)
```

### 4.4 进程锁(可选但推荐)

```cpp
class IProcessLock {
public:
    virtual bool tryLock(const std::string& device_id) = 0;
    virtual void unlock() = 0;
    virtual ~IProcessLock() = default;
};

class IDeviceManager {
    // ...
    virtual void enableProcessLock(const std::string& lock_name = "camera_hal") = 0;
};

// 默认实现:基于 shm_open + pthread_mutex (POSIX) 或 CreateMutex (Windows)
```

### 4.5 Composable Node 与 Intra-process

HAL 本身不感知 ROS,但 ROS Adapter 应该:
- 提供 `Component` 注册(`rclcpp_components::register_node_macro`)
- 暴露 `use_intra_process_comms` 参数
- ROS Adapter 内部使用 `image_publisher` 模式(支持 intra-process 零拷贝)

### 4.6 硬件触发的电气接口

HAL 不涉及电气层面,但要在文档中说明:
- 主从同步线缆需要把 master 的 SYNC_OUT 接到 slaves 的 SYNC_IN
- GPIO 电平要求由设备规格决定
- 如果用 GMSL,所有相机连接到同一 SerDes 即可

### 4.7 ROS Adapter 层的多相机模板

```python
# multi_camera.launch.py 模板
for cam in cameras:
    container.add_composable_node(ComposableNode(
        package='camera_hal_ros',
        plugin='camera_hal::HalCameraDriver',
        name=cam['name'],
        namespace=cam['namespace'],
        parameters=[{
            'serial_number': cam['sn'],
            'sync_mode': cam.get('sync_mode', 'free_run'),
            'enable_process_lock': True,
            ...
        }]))
```

### 4.8 坑

- ⚠️ **Master 必须先启动**:RealSense Slave 在 Master 未启动前不会出帧。Orbbec 同理。HAL 应在文档中说明启动顺序或在 `IDeviceManager` 提供"等所有 master 就绪后再启动 slaves"的协调能力(可选)。
- ⚠️ **同步丢失**:线缆松动导致 Slave 进入 free run,HAL 应通过 `notification_callback` 上报。
- ⚠️ **进程锁泄露**:进程崩溃后 shm 上的 mutex 可能保持 locked。Orbbec 通过 `PTHREAD_MUTEX_ROBUST` 加 `pthread_mutex_consistent` 处理,HAL 实现需照搬。
- ⚠️ **多相机带宽**:USB 3.0 单根总带宽 ~3.2Gbps,3 台 D435 全开会饱和。HAL 不自动处理,但应在 `notification_callback` 上报"USB Bandwidth Exceeded"。
