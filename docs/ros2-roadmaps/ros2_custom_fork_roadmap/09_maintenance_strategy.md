# 09 维护策略 —— 长期维护一个 ROS 2 定制版本

> **核心洞察**：**Fork 不是一次性工程，而是长期承诺**。Fork 的年度成本通常比首次实施成本高 30–60%。本章给出管理这一长期成本的具体机制。

---

## 9.1 成本结构

长期维护一个 ROS 2 Fork 的成本来源：

| 成本项 | 年度占比 | 说明 |
|--------|---------|------|
| **上游追赶**（rebase） | 30–40% | 每次上游发行版升级 |
| **回归测试维护** | 15–25% | 新增功能 / 新 CI |
| **安全漏洞响应** | 5–10% | 上游 CVE 修复跟进 |
| **新需求实现** | 20–30% | 业务驱动的新优化 |
| **文档与培训** | 5–10% | 团队扩张 / 交接 |
| **构建 & 发布** | 5% | CI 基础设施 |

**典型数字**（2–4 个优化阶段 + 10–30 人团队规模）：
- 首次实施：**4–8 人·年**；
- 每年维护：**2–4 人·年**。

## 9.2 分支模型

### 9.2.1 推荐的 Git 结构

```
upstream/main        (ros2/rclcpp 上游)
  │
  └── vendor/<ROS_DISTRO>   (每次取快照，tag 如 vendor/humble-20240601)
          │
          └── our/main      (你的主开发分支，基于 vendor)
                  │
                  ├── feature/events-exec-fix
                  ├── feature/tlsf-allocator
                  └── release/2024.Q2
```

### 9.2.2 两条追赶策略

**A. Rebase 模式**：
- 每次上游发行时把 our/main rebase 到新 vendor tag；
- 冲突解决密集，但历史线性；
- 适合**diff ≤ 3000 行**的情况。

**B. Merge 模式**：
- 上游作为 merge source 合入 our/main；
- 历史有"八爪鱼"合并点；
- 适合 diff 较大、冲突分散的情况。

**经验**：Rebase 模式在 3000 行以内最舒服；超过就要考虑拆分 patch 或改 Merge。

### 9.2.3 Patch 管理

不论哪种分支模式，建议**把每个优化打包为独立 patch 文件**（quilt 风格）：
```
our-patches/
  0001-events-exec-lockfree-queue.patch
  0002-subscription-take-allocator.patch
  0003-executor-static-schedule.patch
  ...
  series
```
好处：
- 每个 patch 有明确的意图 / 收益 / 测试 / 作者；
- Rebase 时可以逐个 apply、逐个解冲突；
- 可以在团队里做 "patch review" 类似 Linux 内核社区的流程；
- 如果某个 patch 上游合了（合入 rclcpp upstream），直接删除此 patch。

## 9.3 上游追赶

### 9.3.1 频率与节奏

- **ROS 2 LTS 发行**（每 2 年）：必须追，规划 1 个季度迁移；
- **ROS 2 non-LTS 发行**（每年）：视业务需要；
- **季度补丁**（sync）：跟，跟得越勤 rebase 越容易；
- **周度 upstream main**：不必跟，但建议每月审阅一次 changelog。

### 9.3.2 追赶清单

1. 拉 upstream tag：`git fetch upstream && git checkout upstream/humble-release`；
2. 打新的 `vendor/humble-YYYYMMDD` tag；
3. 开分支 `rebase/<vendor-tag>`；
4. 逐个 apply our-patches，逐个跑单测；
5. 跑性能基准（阶段 1/4/6 的回归集），对比上一个 vendor 版本；
6. 跑业务端到端测试；
7. 合入 our/main。

单次追赶典型工时：**3–10 人·日**（平均 5 人·日），若某个 patch 因为上游重构无法直接 apply，需要**重写**。

### 9.3.3 如何减少追赶成本

- **保持 patch 小**：一个 patch ≤ 300 行；
- **尽量不改公共接口**：只改私有实现；
- **给上游贡献**：能合入上游的改动就推上去——你的 patch 变成上游代码，追赶成本降到 0。

## 9.4 回归测试体系

### 9.4.1 测试金字塔

```
         ┌─────────────────────┐
         │  E2E 业务回归（小时级）│  每次发行
         └─────────────────────┘
       ┌─────────────────────────┐
       │  性能基准（分钟级）      │  每次 PR
       └─────────────────────────┘
     ┌─────────────────────────────┐
     │  集成测试（分钟级）          │  每次 PR
     └─────────────────────────────┘
   ┌─────────────────────────────────┐
   │  单元测试 + 语法检查（秒级）      │  每次 commit
   └─────────────────────────────────┘
```

### 9.4.2 必须覆盖的场景

- 官方 `test_rclcpp` / `test_rcl` / `test_rmw_implementation` 全套通过；
- 阶段 1–6 每一项优化的性能基线用例（不允许退化 > 5%）；
- 兼容性：至少 2 个真实 robot 业务场景跑 1 小时；
- 健壮性：长时间运行（24h）无内存泄漏 / 无死锁；
- 故障注入：Node kill、网络断、进程 OOM 的恢复行为。

### 9.4.3 性能 CI 的坑

- **构建机器不固定会造成数据漂移**——使用**专用 bench 机器**，隔离 CPU；
- **每次测试都 warm-up**（前 5 s 数据丢弃）；
- **用中位数 + 分位数**，不用平均值（避免异常值污染）；
- **绘制时间序列**：不只看单次，看一个月的趋势；
- **定义 SLO**：P99 < X、P99.99 < Y，违反即 CI 红灯。

## 9.5 版本化与发行

### 9.5.1 内部版本号

建议语义化：
```
<ros2-distro>.<year>.<quarter>.<patch>
例: humble.2024.Q2.3
```

- `ros2-distro`：追赶的上游发行；
- `year.quarter`：每季度一次稳定发行；
- `patch`：hotfix 序号。

### 9.5.2 发行分支

```
release/humble.2024.Q1       <-- 已 frozen，仅接受 CVE / 紧急修复
release/humble.2024.Q2       <-- 当前生产
our/main                     <-- 开发中，下季度发行
```

生产环境应该跑 **release 分支**，不跑 main。

### 9.5.3 变更日志

每个 release 必须有：
- 性能变化对比；
- ABI / API 变化清单；
- 已知问题；
- 升级指南。

## 9.6 团队与知识管理

### 9.6.1 关键角色

| 角色 | 职责 | FTE |
|------|------|-----|
| Fork Owner | 架构决策、上游追赶 | 1.0 |
| rclcpp/rmw engineer | 阶段 2–3 实现 | 1.0–2.0 |
| RT Kernel Engineer | 阶段 6 | 0.5 |
| Performance Engineer | 基准建设 / 回归看板 | 0.5 |
| DevOps / CI | 构建、发行 | 0.5 |
| Release Manager | 季度发行 | 0.3 |

**最少配置**：3 人全职 + 若干兼职。少于此数字时，建议**不要 fork**。

### 9.6.2 知识沉淀

- 每个 patch 必须附 design doc（why + how + 测量数据）；
- Wiki：按阶段 1–6 组织的运维手册；
- Runbook：常见问题（rebase 冲突、性能退化、发行回滚）；
- **交接清单**：新成员入职 4 周能独立处理一个 rebase。

### 9.6.3 与上游社区的关系

- **参加 ROS 2 TSC 会议**：了解上游路线图；
- **订阅 ros2/rclcpp issues**：提前看到可能影响你的变更；
- **把能贡献的 patch 提 PR**：上游合入 = 你的维护成本降低；
- **公开分享优化经验**：社区认可会带来招聘优势。

## 9.7 风险缓解

### 9.7.1 核心工程师流失

**概率**：每 2 年 30–50%。

**缓解**：
- 每个 patch 至少 2 人熟悉；
- 所有 design doc 内部公开；
- 不让任何单独知识只在 1 人头脑里；
- 考虑把关键部分开源（变成社区标准，减少内部 bus factor）。

### 9.7.2 上游重大重构

**典型**：rclcpp 重写 executor（如 Iron 引入 EventsExecutor）、rmw 新增 async API。

**缓解**：
- 提前 1 个发行周期跟踪上游 RFC；
- 准备 fallback 策略（双轨：旧 executor + 新 executor 并存）。

### 9.7.3 业务需求漂移

业务可能在 fork 2 年后说"其实我们不需要这么快了"。

**缓解**：
- 每半年审视一次 fork 的业务价值（参见 [11 决策矩阵](./11_decision_matrix.md)）；
- 若业务需求下降，启动**回退计划**（逐步撤 patch，回到上游默认）；
- 回退比前进代价小——不要固执。

---

## 9.8 什么时候应该放弃 Fork

- 团队规模低于临界（< 3 人）；
- 上游已经合入了你的大部分 patch；
- 业务指标不再卡在 ROS 2 层（瓶颈转到算法 / GPU / 网络）；
- 下一次 rebase 成本预估 > 全部重构上游依赖的成本。

**放弃不是失败**，是理性止损。当初的学习和性能数据仍然是团队的资产。

---

## 9.9 本章小结

- Fork 长期成本 = 首次成本 × (0.3–0.6)，每年重复；
- 分支策略：Rebase 优先、Patch 文件化、按季度发行；
- 测试金字塔必须完整，性能 CI 机器要专用；
- 团队最少 3 FTE；
- 每半年审视一次继续 / 放弃的决策。
